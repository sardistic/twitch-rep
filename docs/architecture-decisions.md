# Architecture Decisions

## ADR-0001: Custom SQL migration runner instead of a migration library (2026-07-18)

Milestone 1 needs an auditable, transactional migration system. A small in-repo
runner (`@chatterscope/postgres`) applies `migrations/NNNN_name.sql` files in
numeric order inside transactions, records them in `schema_migrations` with a
SHA-256 checksum, and refuses to run when an applied file has been modified or
when a file does not match the naming convention. This keeps migrations plain
SQL (matching the handoff's logical schema) with no framework lock-in. If
migration needs grow (down migrations, multiple databases), revisit with
node-pg-migrate or drizzle-kit.

## ADR-0002: ClickHouse schema owned by `@chatterscope/clickhouse` with parameterized TTL (2026-07-18)

Retention must be configurable and TTL support must never be removed. The
schema is generated by `schemaStatements(database, messageRetentionDays)` so
`MESSAGE_RETENTION_DAYS` drives the TTL. `infra/clickhouse/init.sql` is a
convenience copy for first container start using the 365-day default; the
package's `ensureSchema()` is the source of truth.

## ADR-0004: Pilot dashboard is an API-served SPA, not a separate Next.js app (2026-07-18)

The handoff prescribes `apps/web` (Next.js App Router). The pilot deployment
is a single container behind a Cloudflare tunnel; a second frontend container
and framework build was not justified for the current feature set. The
Milestone 5 dashboard (search, profile with role/message filters, notes,
channels, audit) ships as one self-contained HTML page served from the API
(`apps/api/src/web/page.ts`) consuming only the public `/v1` contracts. If the
UI outgrows a single page (routing, SSR, component reuse for the extension),
introduce `apps/web` then — the API contracts are already the boundary.

## ADR-0003: Secrets validated but optional until first use (2026-07-18)

`@chatterscope/config` validates all environment variables with Zod at startup.
`SESSION_SECRET`, `ENCRYPTION_KEY`, and Twitch credentials have no defaults and
are optional in the base schema; each service declares which secrets it needs
via `loadEnv(process.env, { requireSecrets: [...] })` and crashes at startup if
they are missing. This lets Milestone 1 services run without Twitch credentials
while guaranteeing Milestone 2+ services cannot start with missing secrets.
